## 1. Decision Ledger (`decision/ledger.js`)

**Goal**: Catat perjalanan SETIAP kandidat (tidak hanya yang lolos) ke file JSONL append-only.

**File baru**: `decision/ledger.js`
- `addLedgerEntry(entry)` — append 1 line ke `data/decision_ledger.jsonl`
- `queryLedger({ pool, stage, limit, since })` — query historical entries
- `getLedgerSummary(hours)` — aggregasi blocked/deployed/by gate
- `getMissedOpportunities(hours)` — pool yang ditolak (untuk false negative mining)

**File dimodifikasi**:
- `tools/screening.js` — tiap pool yang masuk scoring di-log (lolos/gagal score gate)
- `tools/executor.js` — tiap safety check result di-log (blocked/deployed)
- `index.js` — tiap OOR decision di-log

**Ledger entry format**:
```json
{
  "ts": "ISO timestamp",
  "stage": "screening|executor|management",
  "pool": "pool_address",
  "poolScore": 63.5,
  "finalDecision": "PASSED_SCREENING|BLOCKED|DEPLOYED|HELD_BREAKOUT",
  "blockedBy": ["reason1", "reason2"],
  "wouldDeployIf": ["borderline_recheck_recovery"],
  "ev": 1.8,
  "oorRisk": "MEDIUM",
  "feeVelocity": 85.2,
  "volatility": 2.1,
  "waitMinutes": 0,
  "recheckQueued": false,
  "source": "score_gate|executor_gate|oor_classifier|high_conviction_override"
}
```

**Disk space**: ~500 bytes/entry. 1000 entries/hari = ~500KB/hari. Auto-rotate di 50MB.

## 2. OOR 3-Kategori (`lib/oor_classifier.js`)

**Goal**: Ganti mekanikal timer OOR (22 menit) dengan classifier kontekstual.

**File baru**: `lib/oor_classifier.js`
- `classifyOor(position, opts)` — classify OOR position

**3 Kategori**:
| Klasifikasi | Kondisi | Action | Alasan |
|-------------|---------|--------|--------|
| **OOR_BREAKOUT** | Fee24h >= 7 + waktu OOR >= 10m | `SHIFT_AND_REDEPLOY` (jangan close) | Fee masih sehat, breakout organik |
| **OOR_DUMP** | Fee24h <= 3 ATAU fee drop > 50% | `CLOSE_FAST` | Fee kritis, pool dumping |
| **OOR_NEUTRAL** | Sisanya | `USE_EXISTING_TIMER` (22m) | Tidak cukup data untuk kesimpulan |

**File dimodifikasi**: `index.js` — Rule 3 (far OOR) dan Rule 4 (OOR timer) sekarang pake classifier.

**Key change**: Sebelumnya SEMUA posisi OOR di-close setelah 22 menit. Sekarang:
- BREAKOUT: ditahan (fee masih bagus, jangan close)
- DUMP: di-close lebih cepat
- NEUTRAL: sama seperti sebelumnya

## 3. Fee Velocity Boost (`tools/screening.js`)

**Goal**: Reward pool dengan fee yang sedang naik (boost score), tanpa nambah filter.

**Logic** (di `scorePool` section):
```
fee_change_pct > +20%  → +5 points
fee_change_pct > +10%  → +3 points
fee_change_pct > +5%   → +1 point
fee_change_pct < -20%  → -5 points
fee_change_pct < -10%  → -3 points
fee_change_pct < -5%   → -1 point
```

**File dimodifikasi**: `tools/screening.js` line 1587-1601

**Key feature**: BUKAN gate. Score boost diterapkan langsung ke `pool.pool_score`, sehingga pool dengan fee rising lebih mungkin lolos minPoolScore=56, sementara pool dengan fee dropping dapat penalti kecil.

## 4. Probe Mode (`tools/screening.js` + `tools/executor.js`)

**Goal**: Borderline score (minPoolScore-15 sampai minPoolScore) dapat deploy kecil (0.2 SOL) BUKAN hard block.

**Bagaimana kerja**:
1. **Screening gate** (`screening.js`): jika score < minPoolScore TAPI dalam range borderline (56-15=41 ke 55), pool lolos sebagai `probe_candidate: true`
2. **Executor** (`executor.js`): jika `args.probe_candidate`, amount di-override ke `config.screening.probeSizeSol` (default 0.2 SOL)
3. **Borderline recheck tetap jalan**: pool juga tetap di-queue untuk recheck 2 menit

**Config baru di `config.js`**:
```js
screening: {
  probeModeEnabled: true,   // boolean, enable/disable probe mode
  probeSizeSol: 0.2,        // SOL amount untuk probe deploy
}
```

**Before**: Score 41-55 → hard block + recheck setelah 2 menit
**After**: Score 41-55 → probe deploy 0.2 SOL + recheck untuk evaluasi

## 5. Unified Gate Weight (simplified) (`tools/executor.js`)

**Goal**: High-conviction pools (score >= 70) dapat override minor gate failures.

**Logic** (di `executeTool`, setelah `runSafetyChecks`):
```
if (name === "deploy_position" && safetyCheck gagal) {
  if (poolScore >= 70 && blocking_reason adalah minor gate) {
    → OVERRIDE: lanjut ke deploy
    → Log ke decision ledger sebagai "OVERRIDDEN"
  }
}
```

**Minor gates** (safe to override):
- `bins_below` di bawah minimum
- `bin_step` di luar range (sedikit)
- Fee vs IL proxy (gagal tipis)
- Volatility validation
- Deploy range terlalu kecil

**Major gates** (TIDAK pernah di-override):
- Rugpull / risk_level high
- Wash trading
- Dev sold all / dev blocklist
- Portfolio risk breaker (consecutive losses)
- Max positions
- Wallet heat gate
- Insufficient SOL
- Amount below minDeploy

## Catatan

- Probe mode default 0.2 SOL — aman untuk dry-run. Untuk live, bisa dinaikkan ke 0.3-0.5 SOL.
- Unified Gate Weight override hanya untuk score >= 70. Threshold bisa disesuaikan.
- Fee velocity boost menggunakan `fee_change_pct` dari Meteora API. Jika data tidak available, boost = 0.
- OOR classifier menggunakan `fee_per_tvl_24h`. Jika tidak available, fallback ke OOR_NEUTRAL (existing behavior).
